## 📌 About the App

**Etolv-FLIGHT** is a professional Windows desktop application designed for **travel agencies** and **airline ticketing offices** that need to automate the processing and transmission of flight ticket files across multiple GDS platforms — all from a single unified interface.

### What is a GDS?

A **Global Distribution System (GDS)** is a network that enables travel agencies to access real-time flight inventory, pricing, and booking capabilities from airlines worldwide. The three major GDS platforms are **Amadeus**, **Galileo**, and **Saber**.

### The Problem It Solves

In a traditional workflow, ticket agents must manually handle `.tkt` files generated by the GDS terminal and submit them to the server one by one. This is time-consuming and error-prone — especially when dealing with high volumes of tickets across multiple GDS systems at the same time.

**Etolv-FLIGHT** eliminates this manual process entirely by:

- 👁️ **Watching** a designated folder on your local machine for new ticket files
- ⚡ **Automatically detecting** any new `.tkt` file the moment it appears
- 📥 **Queuing** it for ordered processing without human intervention
- 📡 **Transmitting** it to the correct GDS server automatically
- 📊 **Reporting** the result (success or failure) in real time on the dashboard

The app runs quietly in the background, requiring zero manual effort once configured — letting agents focus entirely on ticketing rather than file management.

## ✨ Key Features

### 🔁 Auto Listener
The app watches a configured local folder in real time using a high-performance file watcher. The moment a new ticket file is added to the folder — whether manually saved or generated by the GDS terminal — the app instantly detects it and starts processing without any user action required.

### 📤 Queue Processing
All detected ticket files enter an ordered processing queue. Each file moves through four clear stages:

| Stage | Meaning |
|---|---|
| ⏳ **Wait** | File is queued and waiting for its turn |
| ⚙️ **Processing** | File is being read and parsed |
| 📡 **Sending** | File is being transmitted to the GDS server |
| ✅ **Response** | Server has responded with a result |

This queue ensures files are processed in order and no ticket is ever skipped or processed twice, even under heavy load.

### 🔐 Secure Login
Access to the app is protected by a username and password login screen. After a successful login, the session is cached securely so the user stays logged in across restarts without needing to re-enter credentials every time.

### 🔄 Auto Updater
The app checks for new versions automatically every time it starts. When an update is available, it downloads silently in the background and prompts the user to restart and apply the update — ensuring everyone always has the latest version with no manual download required.

### 📊 Live Dashboard
Each GDS system has its own dedicated dashboard panel showing a real-time list of ticket files and their current status. Agents can see at a glance which tickets were sent successfully, which are still pending, and which have failed — all without leaving the app.

### 🔒 Single Instance Lock
The app enforces a strict single-instance policy. If a user accidentally tries to open a second copy of the app, it is immediately blocked and a clear message is shown. This prevents duplicate file processing and conflicting server requests.

### ⚙️ Settings Panel
Each GDS panel has its own settings where you configure the specific folder path to watch, along with global settings for the API token, server environment, and window preferences. Settings are saved persistently and survive app restarts.

### 🗂️ Failed File Recovery
If a ticket file fails to send (due to a network error or server issue), the app does not discard it. Instead, it can be moved to a dedicated failed folder for manual review, or the connection can be retried directly from the dashboard with a single click.

### 📝 Error Logging
All application errors and critical events are automatically written to persistent log files on disk. If something goes wrong, the log provides a full trace of what happened, making it easy to diagnose and resolve issues.

### 🎛️ RBAC — Role-Based Access Control
The app supports role-based permissions, meaning different users can have different levels of access. For example, a manager may have full access to all settings, while an agent may only be able to view the dashboard and process tickets.

### 🌙 Minimizable Panels
Each GDS dashboard panel (Amadeus, Galileo, Saber) can be independently minimized, expanded to full view, or closed. This gives users full control over their workspace layout, especially useful on smaller screens or when working with only one GDS at a time.

## 🌐 Supported GDS Systems

Etolv-FLIGHT supports the three major GDS platforms used globally in travel agencies. Each system is fully independent — it has its own folder configuration, dashboard panel, queue, and connection settings.

---

### ✈️ Amadeus

Amadeus is the world's largest and most widely used GDS platform, powering the majority of travel agency bookings globally.

**How the Amadeus module works in this app:**
- You configure a local folder path where Amadeus `.tkt` files are saved.
- The app watches that folder and automatically detects every new file.
- Each file is parsed and placed in the Amadeus queue.
- Files are transmitted to the Amadeus server one by one in order.
- The dashboard updates in real time showing the status of each file.
- If a file fails, it can be moved to a failed folder or retried immediately.

---

### 🌍 Galileo (Travelport)

Galileo, part of the Travelport network, is one of the most widely used GDS platforms in Europe, the Middle East, and Africa.

**How the Galileo module works in this app:**
- Operates identically to the Amadeus module with its own dedicated folder, queue, and dashboard panel.
- Includes a built-in connection error checker that validates the state of the Galileo connection before retrying failed transmissions.
- Failed transmissions can be moved out of the queue and addressed separately without blocking other tickets.

---

### 🌎 Saber

Sabre is a major GDS platform widely used across the Americas and internationally, known for its deep integration with airlines and hotels.

**How the Saber module works in this app:**
- Fully independent folder watcher, queue, and dashboard panel.
- Same robust pipeline as Amadeus and Galileo: detect → parse → queue → send → report.
- Failed tickets can be retried or moved to a separate folder for review.

---

> **All three GDS systems run simultaneously** in the same app window. You can have all three panels open and processing files at the same time without any interference between them.

## ⚙️ How It Works

The following diagram shows the complete lifecycle of a ticket file from the moment it appears in a folder to the final server response:

```
Local TKT Folder
      │
      │  (new .tkt file saved here)
      ▼
 File Watcher
      │  detects the new file instantly
      ▼
 Queue System
      │  file enters the queue with status: WAIT
      │  ──► status changes to: PROCESSING
      │  ──► status changes to: SENDING
      │  ──► status changes to: RESPONSE
      ▼
 Parser
      │  reads the raw file content
      │  extracts ticket data and formats it
      ▼
 API Transmission
      │  sends formatted data to the GDS server
      ▼
 Result
      ├── ✅ SUCCESS → file marked as sent, dashboard updated
      └── ❌ FAILED  → file moved to failed folder or queued for retry
```

**Step-by-step explanation:**

1. **Login** — Open the app and authenticate with your username and password. If you have already logged in before, the session is restored automatically.

2. **Configure** — On first use, open the settings for each GDS panel you want to use and set the local folder path where ticket files will be placed. Save the settings.

3. **Start Listening** — Click the ▶️ play button on the GDS dashboard panel. The app will begin watching the folder immediately. The icon changes to indicate the watcher is active.

4. **Drop Ticket Files** — Ticket files generated by your GDS terminal are saved (or moved) into the configured folder. The app detects them automatically — no manual action is needed.

5. **Auto-Process** — Each detected file is queued and processed in order. You can watch the status change live on the dashboard.

6. **Monitor Results** — Each ticket entry in the dashboard shows its current status. Successfully sent tickets are marked green. Failed tickets are marked red and can be retried.

7. **Pause / Stop** — Click the pause button at any time to stop the folder watcher without closing the app. You can resume listening again at any moment.

## 🚀 Getting Started

This section walks you through the app from first launch to actively processing tickets.

---

### Step 1 — Login

When you open the app for the first time, you will see the **Login Screen**.

- Enter your **Username** and **Password**
- Click the **Login** button
- If your credentials are correct, you will be taken to the **Home Dashboard**
- You can toggle password visibility using the eye icon
- If you make a mistake, click **Clear** to reset the form
- To access app settings before logging in, click the ⚙️ gear icon in the bottom corner of the login screen

> After a successful login, your session is saved. The next time you open the app, you will be taken directly to the Home Dashboard without needing to log in again.

---

### Step 2 — Home Dashboard

After login, the Home Dashboard displays **three GDS panels** side by side:

- **Amadeus Dashboard**
- **Galileo Dashboard**
- **Saber Dashboard**

Each panel shows:
- The current TKT folder path being watched
- Play / Pause button to start or stop listening
- A settings gear icon to configure that GDS
- A live list of ticket files with their processing status

---

### Step 3 — Configure a GDS Panel

Before processing tickets, you must set the folder path for each GDS you intend to use.

1. On the desired GDS panel, click the ⚙️ **gear icon**
2. The settings panel slides open below the panel header
3. Click the 📁 **folder icon** next to "Folder TKT"
4. A folder browser dialog opens — navigate to your ticket folder and select it
5. Click **Save** to confirm

> Repeat this for each GDS system you want to use (Amadeus, Galileo, Saber).

---

### Step 4 — Start Listening

Once the folder path is configured:

1. Click the ▶️ **Play button** on the GDS panel
2. The icon will change to indicate the watcher is now **active**
3. The app is now monitoring the folder in real time

To **pause** listening without closing the app, click the same button again. The watcher will stop but your configuration is preserved.

---

### Step 5 — Process Tickets

From this point, the process is **fully automatic**:

1. A `.tkt` file is saved into the configured folder (by your GDS terminal or manually)
2. The app detects it within milliseconds
3. The file appears in the dashboard with status **Wait**
4. It moves through **Processing** → **Sending** → **Response** automatically
5. On success: the entry is marked ✅
6. On failure: the entry is marked ❌ — you can retry or move the file

---

### Step 6 — Handle Failed Tickets

If a ticket fails to send:

- The dashboard displays the failed entry in **red**
- Click **Retry** to attempt sending again
- Or click **Move to Failed Folder** to remove it from the queue and save it for manual review
- Check the error log if the failure repeats

## ⚙️ Settings & Configuration

The settings panel is accessible from two places:
- The ⚙️ **gear icon on the login screen** — for global app settings
- The ⚙️ **gear icon on each GDS panel** — for per-GDS settings

---

### Global Settings

| Setting | Description |
|---|---|
| **API Token** | The authentication token used to communicate with the server. This is provided to you by your system administrator. Without a valid token, the app cannot send tickets. |
| **Environment** | Switch between **Production** (live server) and **Development** (test server). Always use Production for real ticket processing. |
| **Window Dimensions** | The app remembers your preferred window size and position. You can reset this here if the window appears off-screen. |
| **Photo API** | The API endpoint used for loading profile photos or images within the app. |

---

### Per-GDS Settings (Amadeus / Galileo / Saber)

| Setting | Description |
|---|---|
| **TKT Folder Path** | The full path to the local folder the app will watch for new ticket files. Example: `C:\GDS\Amadeus\Tickets`. This must be set before starting the listener. |

> 💡 **Tip:** Each GDS system requires its own separate folder. Do not point multiple GDS modules at the same folder, as this may cause files to be processed by the wrong system.
